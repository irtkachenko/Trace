# Технічний аудит проєкту Trace

Дата аудиту: 15.03.2026

## 1. Обсяг та методологія
Аудит виконано як статичний code review без запуску застосунку, без виконання тестів і без доступу до продакшн-інфраструктури. Перевірено структуру проєкту, ключові маршрути, бізнес-логіку чату, realtime, storage, помилки, rate limiting, конфігурації Next.js і Supabase.

## 2. Огляд архітектури
- Фреймворк: Next.js (App Router), React 19, TypeScript.
- Data layer: Supabase (PostgreSQL + Realtime + Storage).
- State: TanStack Query + Zustand (presence).
- UI: Tailwind CSS + Radix UI.

Основні контури:
- `src/app` — сторінки, API-роути, layout.
- `src/components` — UI та контейнерні компоненти.
- `src/hooks` — бізнес-логіка (чат, storage, realtime, UX).
- `src/services` — Supabase API/Realtime/Storage.
- `src/lib` + `src/shared/lib` — utilities, errors, perf, rate-limit.

## 3. Сильні сторони
- Добре виділені шари (services/hooks/components).
- Консистентна система помилок і їх обробки (`src/shared/lib/*`).
- Використання React Query для кешу і optimistic updates.
- Реалізовані rate limits на клієнті для ключових операцій.
- Є ретельно продумана UX-логіка для файлів (прев’ю, компресія, optimistic).

## 4. Критичність
Легенда:
- Висока: блокує core-функціонал або створює системний ризик.
- Середня: часткові поломки, деградація UX або стабільності.
- Низька: косметика, дев-режим, документація.

## 5. Знахідки (по пріоритету)

### Висока критичність
1) Підписані URL для приватного bucket можуть не працювати
- Де: `src/services/storage/storage.service.ts` (`getSignedUrl`)
- Причина: `createSignedUrl` викликається без `await`, тому `data`/`error` будуть некоректні.
- Ризик: будь-яке використання `getSignedUrl` дасть неправильну відповідь або падіння.
- Рекомендація: додати `await` та обробку помилок так само, як у `getUrl`.

2) Файли можуть відхилятися навіть при валідних типах, якщо bucket не знайдено
- Де: `src/app/api/storage/config/route.ts`, `src/hooks/useDynamicStorageConfig.ts`
- Причина: коли bucket не існує/не доступний, API повертає `allowedTypes` як розширення (`.jpg`), а клієнт перевіряє MIME через regex — перевірка провалюється.
- Ризик: користувачі не зможуть завантажувати файли у продакшені, якщо storage bucket недоступний або неправильно налаштований.
- Рекомендація: якщо `allowedTypes` не містить `/`, пропускати MIME-перевірку і падати назад на перевірку розширень.

3) Медіафайли можуть “зламатися” після закінчення терміну дії signed URL
- Де: `src/components/chat/MessageMediaGrid.tsx`
- Причина: підписані URL кешуються назавжди, але мають TTL (1 година). При повторному рендері URL не оновлюється.
- Ризик: користувачі бачать “биті” медіа після певного часу.
- Рекомендація: кешувати разом із `expiresAt` і оновлювати URL після TTL, або кешувати на менший інтервал.
